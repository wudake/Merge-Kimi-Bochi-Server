## 一、技术架构总览

### 1.1 架构选型：前后端分离 + 单体服务

考虑到团队规模、开发速度和运维成本，采用 **单体架构 + 前后端分离** 方案：

```
┌─────────────────────────────────────────────────────────────┐
│                        客户端层                              │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────────┐  │
│  │  Web 前端   │  │  平板浏览器  │  │  未来扩展：移动端    │  │
│  │  (React)    │  │ (后续适配)  │  │                     │  │
│  └──────┬──────┘  └──────┬──────┘  └─────────────────────┘  │
└─────────┼────────────────┼──────────────────────────────────┘
          │                │
          ▼                ▼
┌─────────────────────────────────────────────────────────────┐
│                      网关层 (Nginx)                           │
│              静态资源托管 / 反向代理 / API 路由                │
└─────────────────────────────────────────────────────────────┘
          │
          ▼
┌─────────────────────────────────────────────────────────────┐
│                      应用服务层                              │
│  ┌─────────────────────────────────────────────────────┐    │
│  │              后端服务 (Node.js / NestJS)              │    │
│  │  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌─────────┐ │    │
│  │  │ 账号模块  │ │ 内容模块  │ │ 选题模块  │ │ 报表模块 │ │    │
│  │  └──────────┘ └──────────┘ └──────────┘ └─────────┘ │    │
│  │  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌─────────┐ │    │
│  │  │ 询盘模块  │ │ 素材模块  │ │ 用户权限  │ │ 通知模块 │ │    │
│  │  └──────────┘ └──────────┘ └──────────┘ └─────────┘ │    │
│  └─────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────┘
          │
          ▼
┌─────────────────────────────────────────────────────────────┐
│                      数据存储层                              │
│  ┌──────────────┐  ┌──────────────┐  ┌─────────────────┐   │
│  │  PostgreSQL  │  │    Redis     │  │   本地文件系统   │   │
│  │  (主数据库)   │  │  (缓存/会话)  │  │  (报表导出/日志) │   │
│  └──────────────┘  └──────────────┘  └─────────────────┘   │
└─────────────────────────────────────────────────────────────┘
          │
          ▼
┌─────────────────────────────────────────────────────────────┐
│                      外部依赖                                │
│  ┌──────────────────┐  ┌────────────────────────────────┐   │
│  │  工厂内网 NAS     │  │         未来扩展               │   │
│  │  (SMB/CIFS 协议)  │  │   平台API / CRM / ERP 对接     │   │
│  └──────────────────┘  └────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 技术栈选型

| 层级 | 技术选型 | 选型理由 |
|------|----------|----------|
| **前端框架** | React 19 + TypeScript | 生态成熟，类型安全，团队易招聘 |
| **前端构建** | Vite 6 | 极速冷启动，配置简单 |
| **UI 组件库** | Ant Design 5.x | 企业级组件丰富，中文文档完善 |
| **日历组件** | FullCalendar / 自研 | 周视图拖拽核心交互 |
| **状态管理** | Zustand | 轻量，适合中小型项目 |
| **后端框架** | NestJS 11 (Node.js) | 模块化架构，TypeScript 原生，企业级规范 |
| **ORM** | Prisma 6 | 类型安全，迁移管理，开发效率高 |
| **数据库** | PostgreSQL 16 | 关系型数据为主，JSON支持良好，免费 |
| **缓存** | Redis 7 | 会话存储、热点数据缓存、排行榜 |
| **文件存储** | 本地文件系统 | 仅存储报表导出、缩略图；大素材存NAS |
| **身份认证** | JWT (Access + Refresh Token) | 无状态，适合前后端分离 |
| **API 文档** | Swagger / OpenAPI | 自动生成，前后端协作 |
| **任务调度** | node-cron / bullmq | 定时报表生成、提醒通知 |
| **Excel 解析** | xlsx / exceljs | 选题/账号批量导入、报表导出 |
| **PDF 生成** | pdfkit / puppeteer | 报表 PDF 导出 |
| **图表库** | ECharts / Ant Design Charts | 数据看板图表展示 |

## 三、API 设计规范

### 3.1 RESTful API 规范

- **基础路径**: `/api/v1`
- **认证方式**: `Authorization: Bearer <token>`
- **响应格式**:
  ```json
  {
    "code": 200,
    "message": "success",
    "data": {},
    "timestamp": 1714368000000
  }
  ```
- **错误格式**:
  ```json
  {
    "code": 400,
    "message": "参数错误",
    "errors": [{"field": "title", "message": "标题不能为空"}],
    "timestamp": 1714368000000
  }
  ```

## 七、技术风险与应对

| 风险点 | 影响 | 应对方案 |
|--------|------|----------|
| 周视图日历拖拽交互复杂 | 开发延期 | 调研 FullCalendar/antd Calendar，预留1周缓冲 |
| NAS 路径在浏览器中打开受限 | 素材功能不可用 | 主方案：点击自动打开 `smb://` 链接；降级方案：提供「复制路径」按钮，用户自行粘贴到文件管理器 |
| 批量导入格式不统一 | 用户体验差 | 提供严格模板，增加预览和错误提示 |
| 数据量大后报表生成慢 | 性能问题 | 引入 Redis 缓存，报表数据预计算 |
| 多人同时编辑内容冲突 | 数据不一致 | 乐观锁（version 字段）或最后写入优先 |

## 八、待完善事项（已确认，2026-04-29）

### 8.1 需求确认
| # | 问题 | 确认结果 |
|---|------|---------|
| 1 | 多平台一键分发含义 | **仅记录关联账号**，内容由运营人员手动到各平台发布，发布后回填链接。系统不调用任何平台 API。 |
| 2 | NAS 访问方式 | **点击路径自动打开**（smb:// 协议）。前端使用 `<a href="smb://...">` 或 `window.open()` 唤起本地文件管理器。 |
| 3 | 报表导出格式 | **Excel + PDF 两者都要**。周报/月报支持 Excel 导出，看板概览支持 PDF 导出。 |
| 4 | 内容审批流程 | **不需要审批**。运营专员自行维护状态流转：待制作 → 制作中 → 待发布 → 已发布 → 已归档。 |

### 8.2 技术决策
| # | 问题 | 确认结果 |
|---|------|---------|
| 5 | 服务器部署环境 | **本地服务器 / 内网部署**。系统仅在工厂内网访问，不暴露公网。 |
| 6 | 域名和 HTTPS 证书 | **直接使用 IP 地址访问**，暂不配置域名和 HTTPS。内网环境下 HTTP 足够。 |
| 7 | 数据库备份策略 | **每天增量备份 + 每周全量备份**，保留最近 30 天。使用 pg_dump + cron 定时任务实现。 |
| 8 | 账号体系对接 | **不需要 SSO**。系统独立管理用户账号，超级管理员在系统内创建用户。 |

### 8.3 设计相关
| # | 问题 | 确认结果 |
|---|------|---------|
| 9 | UI 设计资源 | **无设计师**，直接使用 Ant Design 5.x 默认风格。通过 ConfigProvider 配置主色即可。 |
| 10 | 数据看板图表风格 | **简洁数字卡片 + 折线图/柱状图为主**。使用 ECharts/Ant Design Charts 实现，白底常规后台风格。 |
| 11 | 平板适配优先级 | **低优先级，暂不处理**。MVP 和 V1.0 阶段仅支持桌面浏览器 1920x1080 分辨率。 |

### 8.4 数据迁移
| # | 问题 | 确认结果 |
|---|------|---------|
| 12 | 历史数据导入范围 | **需要导入账号和选题库**。内容和询盘从零开始录入。 |
| 13 | 历史数据现状 | **数据分散在多个 Excel 和文档中，格式不统一**。需要 1-2 周数据清洗和导入开发时间。 |

### 确认后的关键影响
1. **无需对接任何平台 API**：开发计划中的「平台 API 对接」从 V2.0 可选改为「暂不考虑」，大幅简化架构。
2. **内网 HTTP 部署**：无需配置 Nginx SSL、域名解析、证书续期，部署复杂度降低。但仍保留 Nginx 作为反向代理和静态资源托管。
3. **需要数据清洗导入工具**：在阶段一中增加「账号/选题 Excel 导入」功能，支持多格式兼容和错误提示。
4. **报表导出需支持双格式**：后端需集成 `exceljs`（Excel）和 `puppeteer`/`pdfkit`（PDF）两个库。
5. **smb:// 协议兼容性风险**：部分浏览器（Chrome/Edge 默认）可能阻止 smb:// 链接。需在素材库页面增加「复制路径」按钮作为降级方案。

## 九、开发规范

### 9.1 Git 分支策略（Git Flow 简化版）

```
main        ─────●─────●─────●─────  生产环境
                 ↑
release/v1.0  ───●─────●            发布分支（仅修复 bug）
               ↗
develop   ────●─────●─────●─────●──  开发主分支
             ↗     ↗     ↗
feature/xxx ─●────●     │           功能分支（从 develop 切出）
             │          │
feature/yyy ─●────●─────●           功能分支
```

- `main`: 生产分支，只能合并 release 分支
- `develop`: 开发主分支，功能开发完成后合并至此
- `feature/*`: 功能分支，每个功能/模块一个分支
- `release/*`: 发布分支，提测和 bug 修复

### 9.2 代码提交规范

```
<type>(<scope>): <subject>

<body>

<footer>
```

Type 类型：
- `feat`: 新功能
- `fix`: Bug 修复
- `docs`: 文档更新
- `style`: 代码格式（不影响功能）
- `refactor`: 重构
- `perf`: 性能优化
- `test`: 测试相关
- `chore`: 构建/工具相关

示例：
```
feat(contents): 实现内容周视图日历组件

- 支持日/周/月三种视图切换
- 支持拖拽调整内容排期
- 支持点击创建新内容

Closes #123
```

### 9.3 接口开发规范

1. 所有接口必须包含 Swagger 装饰器文档
2. 所有 DTO 必须使用 class-validator 校验
3. 所有控制器方法必须使用统一响应格式
4. 敏感操作必须记录操作日志
5. 数据库操作必须使用事务（涉及多表写入时）
