# 三项目轻量整合设计

> 文档版本:v1.0
> 日期:2026-05-09
> 范围:Operation_Manage、Facebook-Script-Extract、Dake-Video-Auto 整合方案
> 路线:**轻度整合**(同仓库 + 共享基础设施 + 单子域路径前缀 + 统一 SSO),非深度重构。
> 取代文档:`三项目整合架构设计.md`(已废弃,该文档为深度重构方案,工作量约 6 周;本文档目标 2-3 周)

---

## 1. 关键决策(已与需求方确认)

| 项 | 决策 |
|---|---|
| **整合深度** | 轻度:三个项目代码尽量不改,放进同一仓库,共享 Redis/Postgres/Docker 网络/Python 依赖底层 |
| **域名** | 单子域名 `app.<主域名>`(主域名待定),独立站主域不受影响 |
| **前端形态** | Portal 在根路径 `/`,三个项目分别挂在 `/op/`、`/fbse/`、`/dva/` 路径前缀下,通过 Nginx 反向代理分发 |
| **认证** | 统一 SSO,以 Operation_Manage 的 JWT 为唯一认证源;Nginx `auth_request` 把用户身份透传给后两个项目 |
| **服务器** | 中型,无 GPU(8C/16G+);Whisper 走 CPU + medium 模型 |
| **并发规模** | 小团队(几人),Celery worker 2-3 个即可 |
| **数据保留** | 三个项目的现有生产数据全部保留,需写迁移脚本 |
| **代码源** | 已 clone 到 `/root/Merge/{Operation_Manage, Facebook-Script-Extract, Dake-Video-Auto}` |

**关于子项目的承载方式**有两种,二选一:

| 方式 | 优点 | 缺点 |
|---|---|---|
| **A. Git submodule** | 三个原仓库保持独立维护,改动可单独提交回各自 GitHub 仓库 | 团队需熟悉 submodule 命令 |

**推荐 A(submodule)**,因为后续如果某个项目要单独发布或下线,操作清晰。

## 4. Nginx 路由设计

### 4.1 主配置(`nginx/nginx.conf`核心片段)

```nginx
upstream opman_api    { server opman_api:3000; }
upstream opman_web    { server opman_web:80; }    # 构建后的静态资源容器
upstream fbse_api     { server fbse_backend:8000; }
upstream fbse_web     { server fbse_frontend:80; }
upstream dva_app      { server dva_app:5000; }

server {
    listen 443 ssl http2;
    server_name app.example.com;

    ssl_certificate     /etc/nginx/ssl/fullchain.pem;
    ssl_certificate_key /etc/nginx/ssl/privkey.pem;

    # ───── Portal 静态页(根路径) ─────
    location = / {
        root /usr/share/nginx/portal;
        try_files /index.html =404;
    }
    location /portal-assets/ {
        alias /usr/share/nginx/portal/;
    }

    # ───── SSO 校验入口(内部使用) ─────
    location = /_auth_verify {
        internal;
        proxy_pass http://opman_api/api/v1/auth/verify;
        proxy_pass_request_body off;
        proxy_set_header Content-Length "";
        proxy_set_header X-Original-URI $request_uri;
    }

    # ───── Operation_Manage(/op/) ─────
    include /etc/nginx/locations/op.conf;

    # ───── Facebook-Script-Extract(/fbse/) ─────
    include /etc/nginx/locations/fbse.conf;

    # ───── Dake-Video-Auto(/dva/) ─────
    include /etc/nginx/locations/dva.conf;

    # ───── 登录/登出(走 OPMan) ─────
    location /login {
        return 302 /op/login;
    }
}

server {
    listen 80;
    server_name app.example.com;
    return 301 https://$host$request_uri;
}
```

### 4.2 OPMan(`/op/`)的 location 片段

```nginx
# /op/api/v1/* → NestJS API(已有 globalPrefix=api/v1)
location /op/api/ {
    rewrite ^/op/api/(.*)$ /api/$1 break;
    proxy_pass http://opman_api;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
}

# /op/* → 静态前端(已 build 后由独立 nginx 容器托管)
location /op/ {
    auth_request /_auth_verify;
    auth_request_set $auth_user_id $upstream_http_x_user_id;
    proxy_set_header X-User-Id $auth_user_id;
    proxy_pass http://opman_web/;
}
```

### 4.3 FB-SE(`/fbse/`)的 location 片段

```nginx
# /fbse/api/* → FastAPI(去掉 /fbse/api 前缀)
location /fbse/api/ {
    auth_request /_auth_verify;
    auth_request_set $auth_user_id $upstream_http_x_user_id;

    rewrite ^/fbse/api/(.*)$ /$1 break;
    proxy_pass http://fbse_api;
    proxy_set_header X-User-Id $auth_user_id;
    proxy_set_header X-Original-Host $host;
}

# /fbse/ws/* → WebSocket
location /fbse/ws/ {
    auth_request /_auth_verify;
    rewrite ^/fbse/ws/(.*)$ /ws/$1 break;
    proxy_pass http://fbse_api;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
}

# /fbse/* → 前端
location /fbse/ {
    auth_request /_auth_verify;
    proxy_pass http://fbse_web/;
}
```

### 4.4 DVA(`/dva/`)的 location 片段

```nginx
# /dva/* → Flask(透传 SCRIPT_NAME 让 Flask 知道自己挂在子路径)
location /dva/ {
    auth_request /_auth_verify;
    auth_request_set $auth_user_id $upstream_http_x_user_id;

    proxy_pass http://dva_app/;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-Prefix /dva;
    proxy_set_header X-User-Id $auth_user_id;
    proxy_set_header SCRIPT_NAME /dva;
}
```

### 4.5 路径冲突说明

`/api/` 在三个项目中都有可能出现。本方案用**模块前缀**(`/op/api`、`/fbse/api`、`/dva/api`)显式区分,Nginx 用 rewrite 在转发前剥离前缀,**业务后端代码无需感知**。

## 5. 统一 SSO 方案

### 5.1 选型:Nginx `auth_request`

**原因**:OPMan 已有 JWT,改造量最小。FB-SE 和 DVA 不再实现 JWT 校验,只信任 Nginx 透传的 `X-User-Id` / `X-User-Role` header。

### 5.2 流程

```
1. 用户访问 https://app.example.com/op/dashboard
2. Nginx 触发 auth_request → /_auth_verify
3. /_auth_verify 内部转给 OPMan: GET /api/v1/auth/verify
   带上 cookie 中的 JWT
4. OPMan 返回:
   - 200 + headers (X-User-Id, X-User-Role) → 放行
   - 401 → Nginx 返回 401,前端跳转 /op/login
5. Nginx 把 X-User-Id 透传到目标后端(/op、/fbse、/dva)
6. FB-SE / DVA 信任这个 header,不再自己做 JWT 校验
```

### 5.3 OPMan 需要新增的接口

```ts
// apps/api/src/modules/auth/auth.controller.ts
@Get('verify')
async verify(@Req() req: Request, @Res() res: Response) {
  const token = req.cookies?.access_token ||
                req.headers.authorization?.replace('Bearer ', '');

  if (!token) return res.status(401).send();

  try {
    const payload = await this.jwtService.verifyAsync(token);
    res.setHeader('X-User-Id',   payload.sub);
    res.setHeader('X-User-Role', payload.role);
    res.setHeader('X-Username',  payload.username);
    res.status(200).send();
  } catch {
    res.status(401).send();
  }
}
```

### 5.4 OPMan 现有登录改造点

- 登录成功后:JWT 同时写入 **HttpOnly cookie**(`Domain=app.example.com; Path=/; Secure; SameSite=Lax`),不仅返回 body
- 现有的 Bearer Token API 调用方式保留(向后兼容)

### 5.5 FB-SE / DVA 改造

- **FB-SE**:`api/core/security.py` 删掉 `X-API-Key` 校验,改为读 `X-User-Id` header(由 Nginx 注入,业务直接信任)
- **DVA**:`app_simple.py` 删掉 `USERS` 字典 和 `login_required` 装饰器,改为读 `request.headers.get('X-User-Id')`,无 header 即拒绝

## 6. 共享基础设施

### 6.1 PostgreSQL

只有 OPMan 用 Postgres,FB-SE 和 DVA 都不用,所以 **不需要数据库合并**,只需复用同一个 Postgres 实例即可(为以后扩展预留 schema)。

```sql
-- infra/postgres/init.sql
CREATE DATABASE opman;
-- 预留:未来扩展时用
-- CREATE DATABASE shared_metadata;
```

### 6.2 Redis(三方共用同一实例,通过 DB 编号隔离)

```
DB 0: OPMan(NestJS 缓存、refresh token 黑名单)
DB 1: FB-SE(Celery broker + result + 元数据,原默认 db 0,改成 db 1)
DB 2: DVA(预留,目前不用)
```

FB-SE `.env` 改:
```
REDIS_URL=redis://redis:6379/1
CELERY_BROKER_URL=redis://redis:6379/1
CELERY_RESULT_BACKEND=redis://redis:6379/1
```

### 6.3 Python 依赖(可选优化)

DVA 和 FB-SE 都依赖 `ffmpeg` 系统包、`yt-dlp`、`playwright`。可以共用一个基础镜像减少构建体积:

```dockerfile
# infra/docker/python-media-base/Dockerfile
FROM python:3.11-slim
RUN apt-get update && apt-get install -y \
    ffmpeg curl wget chromium && \
    rm -rf /var/lib/apt/lists/*
RUN pip install --no-cache-dir \
    yt-dlp playwright && \
    playwright install --with-deps chromium
```

DVA 和 FB-SE 各自的 Dockerfile 改为 `FROM python-media-base`,减少 ~500MB 重复层。**这是最直接的"节约服务器资源"收益**。

### 6.4 文件存储

各项目自己的文件挂载点保留,但放到统一的 named volume 下,便于备份:

```yaml
volumes:
  opman_uploads:      # OPMan 素材
  fbse_output:        # FB-SE 视频/字幕产物
  fbse_temp:
  fbse_whisper_cache: # Whisper 模型缓存(大,不可丢)
  dva_videos:
  dva_assets:         # DVA Logo/BGM/TTS,118MB,首次需从原仓库初始化
  dva_static:
  postgres_data:
  redis_data:
```

| 同时保留并清理 `app.py`、`app_multi_user.py`、`app_simple.py`(以 simple 为主,其他归档) |
| `app_simple.py` 中的 `assets/*` 二维码生成 | 输出二维码包含的 URL 必须是 `https://app.{MAIN_DOMAIN}/dva/...`,不能是内网 5000 端口 |

**预估工作量**:3 天(DVA 改动最复杂,因为 Jinja2 模板中可能有写死路径)

### 10.1 Operation_Manage(PostgreSQL)

### 10.2 Facebook-Script-Extract(Redis + 文件)

docker cp fbse_redis:/data/dump.rdb ./fbse_dump.rdb

redis-cli -h <new_redis> --pipe < <(redis-cli --rdb-to-resp fbse_dump.rdb)

## 13. 风险与缓解

| 风险 | 影响 | 缓解 |
|---|---|---|
| **DVA Jinja2 模板写死路径未排查干净** | UI 出现 404 / 链接失效 | 灰度切流量,部署后人工巡检所有功能;保留原服务作为回滚 |
| **Cookie 跨子路径不共享** | 每模块要重新登录 | Cookie `Path=/` `Domain=app.<主域名>`,SameSite=Lax 必须正确 |
| **Nginx auth_request 给 OPMan 增加 QPS** | OPMan 压力上升 | OPMan 内部缓存 verify 结果(如 Redis,5 秒 TTL) |
| **WebSocket 跨子路径** | FB-SE 实时进度断 | Nginx upgrade 头透传配置必须包含,本文档第 4.3 节已给出 |
| **数据迁移期间双写问题** | 迁移期间数据丢失 | 迁移前停旧服务 30 分钟;迁移后立即停旧服务,避免双写 |
| **DVA 的 118MB assets 入仓** | 仓库体积膨胀 | `assets/` 加 .gitignore,通过备份恢复机制初始化 volume |
